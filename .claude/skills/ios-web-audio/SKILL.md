# iOS Safari 網頁音訊開發：實戰鐵律

以下每一條都是在真機上反覆失敗後驗證出來的行為，Apple 文件多半不寫。
本專案（嬰兒哭聲偵測播放器，v1→v33）的完整演進史可佐證每一條。

## 一、路由（哪個喇叭出聲）鐵律

1. **麥克風開著時，頁面的所有播放輸出都會被留在手機**，藍牙/AirPlay 音響收不到任何訊號。
   → 播放前必須「先停掉麥克風軌、suspend 錄音用的 AudioContext，再延遲 ~800ms 才 play()」。
2. **在錄音工作階段中建立的 AudioContext，其輸出會被永久釘在手機喇叭**，之後關麥克風也不會遷移。
   → 播放預設走純 `<audio>` 元素（與 YouTube 同路徑，路由正確）；只有需要即時音量控制（淡出）時才建 MediaElementSource 音訊圖，並向使用者說明「淡出與外接音響不相容」的取捨。
3. **Web Audio 的輸出不走 AirPlay**（WebKit 限制）。AirPlay 用戶必須用純元素播放。
4. **iPhone 上 `element.volume` 是唯讀**（設定被無聲忽略）。程式控音量只能靠 Web Audio GainNode——與第 2、3 條衝突，這是平台級三難，無法全拿。
5. **「播放→錄音→再播放」循環後，路由仲裁會卡死在手機喇叭**，被動等待、延遲加長、先播無聲音軌引導都救不回。
   → 唯一有效解法：**深度音訊重置**——播放結束後把頁面全部音訊資源（mic tracks、所有 AudioContext）`close()` 掉，靜默 ~3 秒，再全部重建。效果等同頁面重載，但保留手勢授權（見下）。
6. **getUserMedia 可能抓到藍牙音響的通話麥克風**，會把藍牙連線切成 HFP（通話模式），高音質 A2DP 回不來。
   → 取得權限後用 `enumerateDevices()` 找內建麥克風（label 比對 /iphone|ipad|built-in/i），之後所有取用都用 `deviceId: { exact: id }` 鎖定。

## 二、授權（能不能出聲、音質好壞）鐵律

7. **使用者手勢是一種「貨幣」**：按鈕點擊授予的完整播放權，`location.reload()` 會全部沒收。
   重載後靠「有麥克風權限」勉強播放的聲音，會被丟進**通話級劣化管線**（變質、碎片循環）。
   → 絕對不要用自動重載當例行手段；要重置音訊就用深度音訊重置（第 5 條）。
8. **每個會被程式播放的媒體元素，都要在手勢內先「靜音播放一瞬再暫停」解鎖**（unlock 模式），
   包括備用元素（如保連音軌）。只解鎖主播放器是不夠的。
9. **媒體管線會無預警卡死**：`currentTime` 凍結、聲音碎片循環、`pause()` 被無視。
   → play() 後 4 秒做健康檢查（`paused || currentTime < 0.3` 即卡死）；卡死唯一解是重載頁面。
   停止按鈕也要加卡死保險（pause 後 700ms 驗證，無效就重載脫困）。

## 三、長時間掛機鐵律

10. **麥克風軌會被 iOS 無預警收走**（播放前後、來電、Siri、切 App）且不自動恢復。
    → 每 3 秒看門狗檢查 `track.readyState` 與 `ctx.state`，掛了就自動重新 getUserMedia。
    → Safari 網站設定將麥克風設為「允許」，重取用才不會跳詢問視窗。
11. **藍牙音響會因「沒收到聲音」自動關機/斷線**——判斷依據是訊號內容，不是連線本身。
    連續播放無聲音軌無效（錄音期間根本到不了音響，見第 1 條）。
    → **脈衝式保連**：每 4 分鐘暫停麥克風 ~1 秒後播 5 秒 17.5kHz 微弱音（人耳聽不到、
    完全在偵測分析頻帶外不會自我觸發、對音響是有效訊號），再自動接回麥克風。
12. 螢幕鎖定/背景分頁會凍結 rAF 迴圈 → `navigator.wakeLock.request("screen")` ＋ 指示使用者
    設「自動鎖定：永不」＋插電＋勿擾模式。

## 四、聲學偵測演算法教訓

13. **自相關法找基頻的經典假象**：平滑訊號在極小 lag 的相關必然趨近 1，會偽裝成「搜尋上界
    頻率」的完美基頻（本案例：安靜房間永遠顯示 F0=1002Hz、r>0.9，讓基頻關卡形同虛設）。
    → 從更小的 lag 開始掃描，等相關**首次跌破 0.3（去相關點）之後**的峰值才算數；
    並排除正好落在搜尋邊界 ±1 的峰值。
14. **絕對 dB 門檻在不同手機上不可靠**（麥克風增益差異大）→ 用「高出背景噪音基準 N dB」的
    相對門檻，基準用快降慢升的追蹤器；開始監聽先做 3 秒快速校準。
15. **背景基準會被靜音期毒化**：播放/脈衝期間麥克風離線，若迴圈照跑，基準會被拖到 -100dB
    以下，重連後一整分鐘「任何聲音都爆表」。→ 麥克風不在線就凍結基準與累積；重連後快速重校 2 秒。
16. 間歇性聲音（哀叫）用**短聲計數**（20 秒內 N 聲）而非持續累積；每聲要有品質門檻
    （≥250ms 且其中 ≥120ms 有清晰週期性）擋掉碰撞/摩擦聲。持續聲（大哭）用累積＋緩慢衰減
    （容忍換氣間隙）。嬰兒基頻 250–1000Hz 與成人 85–255Hz 幾乎不重疊，是最強的分類特徵；
    偵測到成人基頻主動排除。
17. 開啟 `autoGainControl` 幫助收遠處小聲音；`echoCancellation/noiseSuppression` 關閉以保留原始訊號。

## 五、除錯方法論（本次最重要的元教訓）

18. **調啟發式參數前，先建觀測系統**：持久化事件紀錄（localStorage、帶時間戳）＋
    每 10 秒自動診斷行（峰值/基準/各條件通過率/最強週期）。「有聲音但沒觸發」自動留數據，
    每次失敗都變成可分析的證據——從「猜參數」變成「對數據調」，收斂速度差十倍。
19. **頁面上放版本號**（v1、v2…每次改都遞增）：iOS Safari 快取極頑固，沒有版本號會浪費
    大量時間在「其實在測舊版」上。
20. 一個版本只改一組相關的東西，讓使用者的真機實測能歸因。桌機/模擬器測不出以上任何路由與授權行為，**只有真機實測算數**。

## 六、部署備忘

21. GitHub Pages：免費帳號私有 repo 不能用（要轉 public）；Source 選分支＋/(root)；
    初期可指向工作分支，穩定後合併進 main 並把 Pages 切過去（網址不變）。
22. 檔案類資產（使用者的音檔）存 IndexedDB 本機記憶，勿放公開 repo（版權/隱私）；
    提供「下載目前音檔」按鈕解決跨裝置搬運。瀏覽器內剪輯大檔用「位元組比例切片＋
    decodeAudioData」避免整檔解碼爆記憶體（500MB MP3 → 只解需要的一段）；WAV 中段切片
    會破壞檔頭，WAV 一律整檔解碼。
